Headline: hotfix for the v1.5.17 regression that broke saving settings, logging QSOs, and QRZ lookups for anyone with ECHOCAT disabled — plus per-network re-spot templates and the Multi-OTA logging rollout finally crossing the desktop/mobile finish line.

If you're on v1.5.17 and seeing "No handler registered" errors, this is the upgrade.

## Critical — "No handler registered" regression fixed

v1.5.17 shipped a one-line regression that silently broke nearly every IPC channel for users who didn't have ECHOCAT enabled. The audio-restart listener I added for the iOS app's pull-to-refresh button was registered on `remoteServer` directly inside the main app-ready block. `remoteServer` is `null` whenever ECHOCAT is off, so the call threw a `TypeError` — and because Electron silently aborts the rest of an `app.whenReady().then()` block on an unhandled throw, every `ipcMain.handle` registration that came after it never ran.

Net effect for anyone who had ECHOCAT disabled in v1.5.17: save settings did nothing, log-QSO did nothing, QRZ lookup failed, the rig-switcher couldn't change rigs, and a long tail of smaller features just stopped working — all with the same opaque "No handler registered" message in the verbose log.

The fix moves the helper to module scope and re-registers the listener inside `connectRemote()` alongside every other `remoteServer.on(...)` handler. It now only attaches when there's a RemoteServer to attach to, survives a runtime toggle of ECHOCAT in Settings, and the desktop "Reset audio devices" button keeps working regardless.

If you'd already worked around this by going back to v1.5.16 and turning ECHOCAT on before updating to v1.5.17 (a few of you in the Discord found this) — you no longer need to. Plain upgrade from anywhere works.

## Re-spot — per-network defaults + log-popout template-clobber fix

Chris NR9Q reported that whenever he typed a one-off comment into the log popout's re-spot field (e.g. "Rare park! Only 4th activation"), his configured default template was silently overwritten — so the next QSO went out with the wrong text. Two fixes shipped here:

- **Settings → Spots now has four separate re-spot template slots** — POTA, WWFF, LOTA, and DX cluster. WWFF and LOTA fall back to the POTA template when left blank, so existing users keep prior behavior with no migration. (SOTA isn't included because SOTAwatch has no public re-spot endpoint; nothing has changed there.) Banner Logger, the spot-row Log modal, the quick re-spot dialog, and the log popout all now pick the right template by network.
- **Log popout no longer persists per-QSO comments as the saved template.** Per-QSO text is now strictly one-shot — the only way to change a saved template is from Settings → Spots. The "Use default" checkbox auto-unchecks the moment you start typing into the comment field (preserving your custom text for that QSO) and re-checks restore the templated default.

## Multi-OTA logging — full desktop + ECHOCAT-Web rollout

The Multi-OTA logging rework that started a few releases back lands fully here. Activators running 2-fers and 3-fers across program boundaries (e.g. a single park that's both POTA and WWFF, or a POTA park overlapping a LLOTA lake) now log one contact and have it ADIF-tagged correctly for every program at once:

- **Per-program inputs in the log popup.** The activation popup splits the reference field into one row per active program, with auto-prefix and known-park name resolution per row. Type "1234" in the POTA row and "KFF-1234" in the WWFF row — the saved QSO carries both, the spot-row Log dialog cross-references both, and the appropriate re-spots fire on each network's API.
- **SOTA cross-reference for activators.** When an activation is running, SOTA summits whose coordinates fall inside the active POTA park are surfaced in the chip strip so you can tag them without manually copying the summit code. Useful for the "this hill is both a SOTA summit and inside a POTA park" cases that come up more often than you'd expect.
- **LLOTA gap plugged.** Previously a multi-program QSO could land in the logbook without its LLOTA tag when the primary program was POTA or WWFF — phase 1 plugged the gap and phase 4 brought the same behavior to ECHOCAT-Web (so multi-OTA logging from the iOS app and the browser client lands in the desktop logbook with the same field set).

## ECHOCAT — worked-QSO sync stability

A trio of small but compounding fixes to the worked-QSO push path that several iOS users (KO6M, K3SBP, others) hit when their local log got large:

- **1 MB cap on the auto-push payload.** The auto-push of the full worked-QSO set on WebSocket connect was sometimes large enough to choke the iOS WebSocket buffer, which then disconnected with no useful error and the app immediately retried — producing the reconnect loop that a couple of you reported during long activations. The desktop now caps the auto-push at 1 MB worth of JSON (chronologically newest first) and the iOS app uses a follow-up paginated fetch for older history if it needs it.
- **Cached JSON + per-client dedupe.** The desktop now caches the serialised worked-QSO blob between sends and dedupes by client. Hitting the iOS Refresh button or having two browser tabs open no longer re-serialises and re-sends the same data multiple times — useful on Raspberry Pi-class hosts.
- **Better disconnect diagnostics.** When a WebSocket closes, the verbose log now records the close code, reason string, and `bufferedAmount` at close time. The reconnect-loop reports were almost impossible to diagnose previously because the desktop logged "[Echo CAT] Client disconnected" with no further context — these three values usually pin down whether it was a network drop, an iOS background-suspend, or a payload-size kill in a couple of seconds.

## Known issue — Scottie 2 vertical red cast

A few users have reported a red cast that appears around row 10-15 of received Scottie 2 images and continues through the rest of the decode. We're investigating — current suspects are the running-median `lineStart` window settling and the continuous-AFC `freqOffset` convergence both finishing in roughly that line range. If you have one of the affected decodes (saved PNG from the SSTV gallery, or even better the source audio), drop it in the Discord — I can sweep the relevant constants against the actual signal and patch it precisely rather than guess.

**Tailscale users on macOS:** if Settings → ECHOCAT says "Tailscale not detected" but you have it installed, you may need to enable MagicDNS in your [admin DNS console](https://login.tailscale.com/admin/dns). The status line tells you the exact toggle to flip and links straight to the page.

**macOS users on Apple Silicon:** the DMG is ad-hoc signed (no paid Developer ID yet). If you see "POTACAT is damaged and can't be opened," run this once from Terminal and then relaunch:

```
sudo xattr -dr com.apple.quarantine /Applications/POTACAT.app
```
