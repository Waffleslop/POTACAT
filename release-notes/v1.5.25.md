# TX Without DAX

**Headline:** on "SmartSDR Direct", POTACAT now transmits FT8 *and* voice straight to the Flex over VITA-49 — the **DAX program is no longer in the path at all**. RDP killed your DAX session? Doesn't matter. Audio goes desktop → Flex over the network, period. Plus: a real fix for an FT8 cycle-timing bug that's been silently truncating every TX since the engine was rewritten, a new DX cluster filter by spotter region/zone, and triage fixes for several Discord reports.

## SmartSDR Direct now covers TX too

The "SmartSDR Direct" audio source was RX-only — POTACAT received audio over VITA-49 but TX still routed through the Windows `DAX Audio TX` device and the DAX program. That asymmetry was the source of the "I'm remote and DAX broke again" failures: RDP audio redirection, a stopped DAX program, a stale Windows device ID — any of those silently killed TX even though PTT still keyed the radio.

v1.5.25 adds the symmetric TX path:

- **FT8/JTCAT** — pre-rendered cycle audio gets packed into VITA-49 `dax_tx` packets and paced to the radio at real-time 24 kHz. No `setSinkId`, no Windows audio device, no DAX program. The FT8 envelope still starts at slot+500 ms (WSJT-X convention).
- **SSB / FM / AM voice from the iOS app** — incoming WebRTC mic audio is downsampled to 24 kHz in an AudioWorklet, forwarded chunk-by-chunk via IPC, and streamed as VITA-49 packets to the radio. The Windows DAX TX device is muted in this mode so the audio doesn't double-feed.

What you can do now that you couldn't before:
- Stop the DAX program entirely if you only use POTACAT (and tools that don't need DAX). Both RX and TX work without it.
- Operate fully via RDP. No more "DAX died, walk to the shack" cycles.
- Run POTACAT on a headless box at the shack without installing SmartSDR's full DAX stack at all (driver still required for the device to exist as a fallback).

You stay on the old (DAX-program) TX path if your audio source isn't set to **SmartSDR Direct**, so this change is transparent for non-Flex users and Flex users on plain DAX.

## FT8 cycles no longer chop themselves off

A separate bug that the SmartSDR Direct work uncovered: the FT8 engine's safety timeout (`(cycleSec - 1.5) * 1000` = 13.5 s) was *always* tighter than the time it took for the renderer to finish playing the 12.64 s audio envelope plus the 200 ms IPC delay plus its own scheduling latency — total ~13.6–13.7 s. PTT was being released ~150–500 ms before the FT8 envelope finished, every cycle. Receivers got a truncated waveform that couldn't be decoded, so callers saw zero replies and zero PSK Reporter spots.

Widened the safety to `(cycleSec - 1) * 1000` = 14 s. ~860 ms of headroom past audio end, 1 s of back-end window before the next slot. This was a stealth fix benefiting *every* FT8 user, not just SmartSDR Direct.

## TX output device now logs by name

A related diagnostic: the TX audio output device label is now resolved and logged to the CAT panel every cycle ("DAX Audio TX", "USB CODEC", "Speakers (Realtek...)", etc.), and `setSinkId` failures now surface a warning instead of disappearing into the renderer console. The device label resolution is cached, fire-and-forget *after* `source.start`, so it can't eat into the FT8 timing budget the way the first attempt did.

## DX cluster: filter by spotter's region (continent / CQ / ITU)

For F4HXJ and anyone tired of seeing cluster spots from regions whose propagation doesn't reach them: **Settings → DX Cluster → Spotter origin filter**.

- Six continent checkboxes (AF / AS / EU / NA / OC / SA — all on by default).
- Optional CQ-zone and ITU-zone text fields that accept comma-separated values and ranges (`14,15,33` or `1-10,27-30`). Empty = no zone filter.
- Filters on the *spotter's* origin (the station who heard them), not the spotted station — the cross-cut you want when you only care about propagation paths that reach your QTH.
- Unknown spotter callsigns always pass through, so odd `/p` / `/qrp` / non-cty.dat-resolvable stations don't disappear.
- Cluster spots only. POTA / SOTA / RBN / PSKR are unaffected.

## Other fixes from Discord triage

- **SSB spot click no longer snaps your filter to 2.4 kHz** (N5WBL): when you click an SSB spot and were already in an SSB-class mode, POTACAT now keeps your live filter setting instead of overwriting it with the saved `ssbFilterWidth` default. Crossing categories (CW → SSB, SSB → DIG) still applies the per-mode default.
- **Hotkey diagnostics in the CAT log** (bjh): chorded `keydown` events now log to the CAT log panel (not just devtools) — `[Hotkey] keydown "Ctrl+F1" → tune` or `[Hotkey] keydown "Ctrl+F1" (no binding configured)`. The `tune` action surfaces an explicit error when its frequency parameter is empty or unparseable. If a custom hotkey isn't working, the CAT log now tells you exactly why.
- **macOS Upgrade button no longer "flashes and does nothing"** (WZ1H): our DMGs are ad-hoc signed (no paid Developer ID), so `electron-updater` can't install over them. macOS now skips the auto-update path entirely and shows the same "Download from GitHub" link the portable builds get. Any failed update on any platform now shows a toast with the GitHub release link instead of resetting the button silently.

## sBitx / Raspberry Pi: ALSA smoke test script ships with the arm64 build

HA3HZ: `scripts/test-alsa.js` is now bundled in the AppImage so you can list ALSA devices and capture a few seconds to a WAV directly from the unpacked app:

```
./POTACAT-1.5.25-arm64.AppImage --appimage-extract
cd squashfs-root/resources/app.asar.unpacked
node scripts/test-alsa.js                    # list ALSA devices
node scripts/test-alsa.js hw:1,1 5 out.wav   # 5-second capture → out.wav
```
