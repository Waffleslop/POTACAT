A bigger release than the version number suggests — the headline is a fundamental rethink of the POTA.app integration after diagnosing why sync was failing for everyone, plus a stack of CW / paddle / ECHOCAT fixes from user reports.

## POTA.app — Reframed as Profile Display

After multiple users (KD2TJU, WD4DAN, and others) hit recurring 403s on sync that no auth fix could resolve, we finally diagnosed the underlying cause: **the `/user/stats/parks/csv` endpoint requires AWS SigV4 signing, not a JWT.** Confirmed by the actual response body once we logged it. POTA.app's API isn't published for third-party apps — Dan WD4DAN, who's on the POTA dev team, confirmed publicly that real third-party API keys are coming with **POTA v3** (timeline: probably next year).

So we've rethought the integration:

- **Worked parks now come from your own QSO log.** A new harvester walks your ADIF log on every load, pulls every `POTA_REF` and `SIG=POTA + SIG_INFO`, and feeds them into POTACAT's worked-parks list. This is more accurate than pota.app's CSV anyway — every QSO you've logged counts as a worked park immediately, no waiting for pota.app to ingest your upload. Walt KK4DF's 12,933-QSO N3FJP import now drives "hide worked parks" / "new park" filtering correctly without any pota.app sync.
- **Profile sign-in still works** — but it now hits the *public* `/stats/user/<call>` endpoint (no auth at all) and just displays your activator / hunter counts, awards, and endorsements in the Cloud settings panel. The legend reads "POTA.app Profile" instead of "POTA.app Sync."
- **Auto-sync, scheduled pulls, and CSV downloads are removed.** No more recurring 403 errors, no more "reconnect to capture a fresh token" red banners. The Verbose log used to fill with `[pota-sync] pull failed` lines on every interval — that's gone.
- **One-click activation upload to POTA.app** — each Past Activation row now has an "Upload to POTA.app" button next to "Export ADIF". Click it and POTACAT writes the activation ADIF, opens pota.app's upload page in your browser, and reveals the file in your file manager so you can drag-drop it onto the upload UI. Works without any private API access, which is the only sanctioned path until v3 lands.

## ECHOCAT — Critical fixes

- **FT8 view no longer kills the WS handler.** A stale `ft8-cycle-indicator` reference in `remote.js` threw `TypeError: Cannot set properties of null` on every FT8 cycle tick, halting incoming WebSocket message processing for the rest of the session. Caught by the v1.5.8 fatal-error banner — first real validation of that diagnostic.
- **Error banner is now actionable.** Selectable text (works on touch with tap-and-hold) plus **Copy** and **Reload** buttons. Footer prompts emailing the message to casey@potacat.com. KM4CFT had to hand-transcribe the v1.5.8 error before this fix — never again.
- **CW sidetone delay on first macro tap, fixed.** iOS Safari and Android Chrome both have ~1–3 seconds of cold-start latency on `AudioContext.resume()`. Anything scheduled in that window gets silently dropped, which ate the first part of "CQ CQ CQ DE …". Standard fix: schedule a 1-sample silent buffer-source play right after `resume()` to force the audio thread to spin up. KM4CFT report.
- **CW paddle dits / dahs no longer hit cold-start.** A perpetual zero-gain anchor oscillator now runs in the CW audio context for the life of the page. Macros warm the engine through their multi-second pre-scheduled sequence, but a single brief paddle dit was too short and the audio thread kept dozing between events. The anchor keeps the engine continuously warm. KM4CFT report on TinyMIDI paddle.
- **Paddle keydown drops are now logged.** When the desktop hasn't enabled remote CW (`cwAvailable` is false), paddle key events were silently swallowed — same symptom as a broken WebSocket. Now prints `[CW paddle] Ignoring key — cwAvailable is false` to the phone's console so the user can see exactly why their paddle is dead.

## Bug Report Flow

- **Banner no longer overlaps the Table View menu.** A `ResizeObserver` watches the bug-report banner and pushes the body content down by its current height while it's visible — multi-line wrapping on narrow windows handled correctly. Filters and rig controls stay clickable while bug-reporting.
- **Verbose Log state is restored on dismiss.** Previously, opening Report a Bug forced verbose logging on but never restored the prior state — users were stuck with the log panel visible after copying the report. Now: snapshots `priorVerboseLog` and `priorLogPanelHidden` at flow start, only reverts what the bug-report flow itself enabled, defensive against the user manually toggling mid-flow.

## Spots Table

- **Pinned tuned spot** — clicking a row to tune now keeps that row visible regardless of age, source toggles, band/mode/region filters, and hide-worked. Works the case where a weak POTA station's spot times out from pota.app mid-QSO and the operator was forced to log manually because the row had vanished. (N5SKT report.) Pin auto-clears when you click a different spot or restart the app.
- **Tuned row stays centered** — `render()` re-centers the pinned row in the viewport only when new spots reflow it off-screen (won't fight users who've scrolled away to look at something else).

## FreeDV

- **DATA-mode opt-out for older Yaesu rigs.** The FTDX3000 caps DATA-mode IF bandwidth at 2.4 kHz — too narrow for FreeDV's ~3 kHz needs. Settings → Spots now has a **"Switch radio to DATA mode (DIGU/DIGL) for FreeDV"** checkbox. Default on (preserves IU7RAL's v1.5.7 fix); off keeps the rig in plain USB/LSB so older Yaesus get the wider IF and you route USB CODEC audio at the rig level. (AB9AI report.)

## Cloud Sync — Bulk Upload

- **Retry on transport errors during initial bulk upload.** Walt KK4DF's 12,933-QSO N3FJP import was dying mid-stream with `ECONNRESET` and `SSLV3_ALERT_BAD_RECORD_MAC` — both flaky-mid-stream connection failures from middleboxes idle-killing keep-alive sockets. Each chunk now retries 3 times with exponential backoff (1s / 3s / 9s) and chunk size dropped from 500 → 250 to reduce per-request time. Terminal failure message tells you exactly which chunk died and that re-running the upload will dedupe — server-side dupe detection has been there all along, this just surfaces it.

## Rig Control

- **rigctld spawn args now in the Verbose log** — `rigctld spawn: -m 311 -r COM5 -s 19200 ...` prints when CAT connects so users (and bug reporters) can confirm the PTT-Port and other flags actually reached hamlib. Solves the "did my Settings save take effect?" diagnostic gap.
- **No more spurious dit at app launch on DTR-keyed rigs.** When the OS opens a serial port, it asserts DTR for ~5–50 ms before our user-space drop runs; on a radio with `USB Keying (CW) = DTR` set, that brief assertion *is* a dit on the air. The dedicated CW key port now opens **lazily** on first paddle event or first CW text-send instead of at startup, so the unavoidable open-time DTR pulse merges into a moment the operator is already intentionally keying. (WD4DAN report.)

## POTA.app Profile (cosmetic)

- **Signed-in label now shows your callsign**, not the Cognito UUID. Dan WD4DAN saw "Signed in as be2176c0-606d-491e-93f3-adfd699a04b3" instead of "WD4DAN" because the JWT label decoder didn't know about pota.app's custom `pota:callsign` claim. Now it does.

## Welcome screen

- **Auto-SSTV blurb fixed** — "Automatically switch to the 40m & 20m SSTV calling frequencies and **receive and decode** images when the radio is idle" (was "download images," which read like POTACAT was pulling from somewhere remote).

**macOS users on Apple Silicon:** the DMG is ad-hoc signed (no paid Developer ID yet). If you see "POTACAT is damaged and can't be opened," run this once from Terminal and then relaunch:

```
sudo xattr -dr com.apple.quarantine /Applications/POTACAT.app
```

**Intel Mac users:** build from source for now — a universal DMG is coming in a later release.
