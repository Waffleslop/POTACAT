# Hamvention Hotfix: FlexRadio CW Actually Transmits Now

Headline: every Flex user's CW macros — from POTACAT desktop, from ECHOCAT iOS, from the CW bar in the popout — have been silently doing nothing for the last seven releases. Fixed in this build alongside ECHOCAT iOS getting audible CW sidetone during TX, the audio-health detector no longer false-firing during long FT8 cycles, and an FT8 timing tightening that improves QSO completion when responders are weak.

If you've ever sent a CW macro on a Flex and wondered why the radio didn't transmit — that wasn't user error. This is the fix.

## SmartSDR CW silently dropping every macro (Casey K3SBP / all Flex users)

Commit `fc578e1` (a few releases back) shipped `sendCwText` in `lib/smartsdr.js` using `cw send "..."` — but the SmartSDR API command is `cwx send "..."` (with the `x`). The Flex radio drops unknown commands silently and returns a non-zero status code that POTACAT's command parser doesn't surface to the verbose log. So every CW macro from a Flex user logged the line `[SmartSDR] CW send: <text>` and produced zero RF, with no error message, for v1.5.13 through v1.5.19.

I noticed this evening while testing the WinKeyer priority fix — sent a "?" expecting `..--..` and the radio sat there silent. The fix is one character (`cw send` → `cwx send`) plus restoring the historical space-to-0x7F encoding that the original `cwx send` callsite was using. The Rig Commands tab's UI default for "CW Text" had the same typo and is also corrected.

If you've stopped sending CW macros on your Flex because they "felt unreliable" — please try again on this build. They work.

## CW sidetone for the ECHOCAT iOS audio bridge (Casey K3SBP)

When the desktop fires a CW macro and you're listening on the iOS app's audio bridge, the bridge has been going silent for the entire macro. The Flex slice receiver is muted by firmware during TX (correct behavior — receiving your own carrier through the slice's RX is just broken audio), and the `dax_rx` stream POTACAT subscribes to goes silent with it. Other rigs have similar gaps depending on USB-codec routing.

The obvious "just turn on the Flex CW Monitor" approach doesn't actually work here — the MON output routes to a different Flex audio stream than `dax_rx`, so flipping it server-side wouldn't reach the bridge anyway.

The fix is rig-agnostic: POTACAT now synthesizes Morse audio locally using Web Audio API oscillators and mixes it into the WebRTC sender's destination *after* the existing TX-mute gain. The bridge stays muted to leaking RX audio (the listener doesn't hear their own carrier feeding back) but the synthesized sidetone reaches the phone speaker. Per-element oscillators with a 5 ms attack/release envelope so the iOS speaker's small driver doesn't render key-clicks.

The pitch is configurable from **Settings → CW Keyer → Pitch (Hz)** — same field that already drove the desktop's local sidetone, just now wired through to the iOS bridge too. Defaults to 600 Hz to match the Flex CW default. Clamped to 300–1200 Hz.

Non-Flex setups (Yaesu/Icom/Kenwood) need a separate mix path into their local audio capture stream — that's a follow-up. For now, Flex users on ECHOCAT get audible CW sidetone for the first time.

## Audio-health detector no longer false-fires during FT8 (Casey K3SBP)

Several FT8 sessions in the last week ended with "Audio bridge keeps failing — Auto-recovery paused" as an iOS toast. The detector at `renderer/remote-audio.html` watches the rig audio coming out of the bridge and flags `peak-zero-while-rx` after five seconds of silence. The flag triggers an iOS auto-restart; three failures in a minute trips the circuit breaker and the user sees the toast.

Problem: the detector didn't gate on whether the rig was transmitting. FT8's 12.6-second envelope, CW macros longer than five seconds, and voice macros all caused the rig audio path to legitimately go silent — and the detector saw five seconds of zero peak and faithfully reported "silent." Every FT8 TX cycle triggered two false restarts before the cycle finished.

Fixed by introducing a unified `_isEffectivelyTransmitting()` predicate that ORs:

- `_remoteTxState` (user PTT / FT8 / voice macros / SSTV — already wired through `handleRemotePtt`)
- A new `_cwTxLockoutUntilMs` timestamp set by `sendCwTextToRadio` for the estimated morse playback duration

Every consumer of "is the rig TX'ing right now" — the VFO popout TX banner, the WebRTC bridge TX-mute, the iOS app's `status.txState`, and the audio-health detector itself — now reads through this one predicate. Adding any future TX source only needs to flip one input.

The detector also gets a 1500 ms post-TX grace window so the rig's audio path can settle (DAX RX hand-off, USB-codec AGC) without false-firing on the first tick after PTT release.

## FT8 late-tap window — WSJT-X-strict slot alignment (Casey K3SBP)

If you've ever tapped an FT8 spot more than half a second into a slot and watched the responder fail to come back, this one's for you.

`MAX_LATE_MS` in `lib/ft8-engine.js` controls how late into an FT8 slot POTACAT will still fire TX before deferring to the next own-phase slot. It was 2500 ms (with intermediate values of 1500 ms and 2500 ms over the last few weeks, trading receive-side margin for click-to-TX ergonomics). The problem: when `msIntoCycle > 500`, the leading-silence pad at `renderer/app.js:18322` disengages and the audio envelope starts at the actual click time, not slot+500ms. The on-air DT at every receiver becomes `(msIntoCycle − 500)` ms, and the trailing PTT-off bleeds into the *next* slot — the responder's slot — clipping the first 100–300 ms of *their* Costas array.

Net effect: QSOs that started with a late tap had measurably lower completion rates because the responder's first reply got partially decoded or missed entirely.

`MAX_LATE_MS` is now 500 — matching WSJT-X's slot-boundary-aligned behavior. Audio envelope lands at exactly slot+500 ms every time. DT=0 at well-tuned receivers. Envelope ends at slot+13.14 s with 1.71 s of clear air before the next slot starts. No bleed into the responder's RX.

Trade-off: taps landing more than 500 ms after a slot boundary now wait for the next own-phase slot (up to ~30 s if the QSO is locked to even-only or odd-only). This is the same wait WSJT-X users live with, and it's the right call: a 30 s delay before a clean transmission beats a 1 s delay before a transmission that breaks the responder's decode.

## ECHOCAT iOS WebSocket close (1009) diagnostics (Walt KK4DF)

Walt reports iOS connecting and immediately disconnecting with WebSocket close code 1009 ("Message Too Big") even with worked-qsos correctly skipped. The leading suspect is the `auth-ok` message carrying the full `_remoteSettings` object — `sstvTemplates`, `customCatButtons`, `remoteCwMacros` on accounts with heavy customization can push this past iOS's WebSocket receive cap.

Two changes shipped to investigate and start mitigating:

- **`perMessageDeflate` enabled on the WebSocket server.** iOS React Native's WebSocket negotiates the `permessage-deflate` extension by default since RN 0.62, and JSON payloads typically halve on the wire. If Walt's `auth-ok` is borderline-over the iOS receive cap, this alone may fix him without further protocol changes. Reasonable defaults (level 6, 1 KB threshold, no-context-takeover for memory bounds on RPi-class hosts).
- **`auth-ok` size measurement in the diagnostic log.** The existing `Initial payload sizes` log already covered every initial-state message except `auth-ok`. Now it includes both the full `auth-ok` payload size and a separate breakout of just the `settings` sub-object, so if 1009 still fires the offender will be unambiguous in his next log.

Walt — please grab this build and reconnect. If you still get the toast, send the new `Initial payload sizes` log line (it'll have `auth-ok=N   └ settings=M` at the front) and we'll target the actual oversized message.
