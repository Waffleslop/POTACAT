A bigger-than-usual point release. Headline is a sweeping SSTV decoder overhaul that lifts low-SNR PSNR by 15-22 dB across all five modes — actual readable images at the noise levels HF reception lives in. Plus a new experimental Icom Network connection type that talks RS-BA1 over UDP for IP-native Icoms (IC-705, IC-9700, IC-7610) and for any radio bridged through wfview's headless `wfserver`.

## SSTV — noise immunity overhaul

K3SBP received two Robot 36 SSTV transmissions on 4/28 that decoded to colored lines, no recognizable image. That kicked off a multi-pass decoder rewrite. Four stacked changes:

* **Band-limited Hilbert + matching band-pass FIR** (passband 500-3500 Hz). The old wide-band Hilbert let broadband noise dominate the freq-estimator's variance; restricting the analytic signal to the SSTV tone band before `atan2` cuts that noise by ~13 dB. Both filters are linear-phase so pixel edges stay sharp.
* **Per-pixel in-band gating + median extraction.** `extractChannel` now drops samples outside [1450, 2350] Hz (sync/sep tail leak) and takes the median of the middle 50% trimmed window — unbiased by residual boundary samples and noise spikes that the trimmed mean was averaging into pixel values.
* **Robot 36 chroma bilateral temporal interp** with per-pixel Y-similarity gating. When line N arrives, line N-1 is re-rendered using averaged Cr/Cb from same-type neighbors at lines N-2 and N. At sharp vertical chroma transitions (think hard-edge color zones in horizontal-stripe images) the bilateral averaging is replaced per-pixel by whichever neighbor's Y is closer to line N-1's, so we don't smudge two unrelated colors together.
* **Adaptive sync-bias calibration** per mode against the new filter chain — Martin +3, Robot 36 -3, Robot 72 +3 samples, Scottie unchanged at -8.

Measured impact at 15 dB SNR on color bars (Martin / Scottie / Robot36 / Robot72): 7.9 / 8.0 / 6.4 / 6.2 dB → **30.2 / 29.7 / 22.9 / 23.6 dB**. At 10 dB SNR: 6 dB → 19-25 dB. Trade-off: slight (~3 dB) regression on absolutely-clean Scottie color bars from the BP-Hilbert narrowing — invisible above 40 dB PSNR, dominated by the 15-22 dB low-SNR gain.

A new `scripts/test-sstv-quality.js` benchmark exercises bars / horizontal-stripe / diagonal / pseudo-photo patterns plus AWGN and clock-drift sweeps so future tuning catches regressions.

## Icom Network (RS-BA1) — experimental connection type

New radio type in Settings → Rig: **"Icom (Network / RS-BA1) — experimental"**. POTACAT desktop now speaks RS-BA1 over UDP, the protocol used by:

* IP-native Icoms — IC-705, IC-9700, IC-7610, IC-7851, IC-R8600 — connecting directly to the radio over LAN, no host PC bridge.
* `wfserver` (wfview's GPLv3 headless server, runs on a Pi/Linux/Mac) bridging USB-only Icoms (IC-7300 / IC-7300 MK II / IC-7100) onto the network.

Phase 1 is **CAT only** — frequency, mode, text-CW. No audio yet (Phase 2). Per-element paddle CW is hardware-only on the IC-7300 family regardless of remote stack — that's a kernel + radio-firmware constraint, not a POTACAT one. CW text macros (CI-V 0x17 / hamlib `send_morse`) work fine remotely.

Settings: host, control port (default 50001), civ port (auto), username, password — username/password come from whatever you set in `wfserver --setup` or the radio's own Network User menu. The transport handles the full handshake (AreYouThere → IAmHere → AreYouReady → Login → Token → ConnInfo → Status → civ-stream sub-handshake), passcode-obfuscates credentials per the protocol, runs ping + idle keep-alives, and renews the auth token every 60 seconds.

This is genuinely experimental — first contact with real radios will surface bugs. KM4CFT and a handful of others are testing first; if you have one of the IP-native Icoms or a `wfserver` setup and want to try, set the connection type, watch the verbose log for `[rsba1/control]` and `[rsba1/civ]` lines, and send the log to me when something breaks. A standalone `scripts/rsba1-probe.js` is included for protocol-level reverse-engineering work.

## Settings popover stops growing off-screen

Reported on Linux, replicated on Windows and likely Mac too. Click Settings → quick-settings popover appears → moments later the audio-device fields populate with verbose USB device names → the popover stretches leftward and the left edge of the menu pushes off the screen, with switches and buttons clipped and unclickable.

Fix: panel is now locked at 280px wide, with `max-width: calc(100vw - 16px)` as a safety on narrow windows. The audio-device select boxes are forced to shrink-fit via `width: 0; flex: 1 1 0` (Chromium ignores `min-width: 0` on native form controls in flex layouts, which is what allowed them to push the parent wider). Long device names truncate cleanly with an ellipsis when the select is closed; the open dropdown still shows full text.

## Worked Parks — surfaced where users actually look for it

KE4WLE asked how to get his external log history into POTACAT now that worked-parks comes from your QSO log instead of pota.app sync. Import Log existed (Settings → Logging → Import Log) but there was no signpost connecting "I want hide-worked-parks to work" with "this is the button." Three changes:

* New **"Import existing logs..."** button in Settings → Spots → POTA, directly under "Hide already-worked parks", with explanatory help text naming the supported source apps (N3FJP, HRD Logbook, HAMRS, Log4OM).
* Same button in the Quick Settings dropdown next to the Hide Worked Parks toggle, for one-tap access.
* Import-result dialog now leads with the worked-parks bridge: *"12,933 QSOs imported. Worked-parks list now has 1,247 references (12 new from this import)."* No more wondering whether the import actually wired into the filter.

## CW paddle — clearer UX when DTR keying isn't reachable

KM4CFT on Linux + IC-7300 paddle-on-phone via ECHOCAT. Linux's cdc_acm USB-CDC driver rejects the `TIOCMSET` ioctl that node-serialport (and most other stacks) use to flip DTR for CW keying. Symptom: phone plays sidetone, radio silent, user assumes ECHOCAT is broken.

Now: when the transport hits that rejection, it emits a `pin-unsupported` event; the phone receives a `cw-paddle-available: false` flag and **suppresses local sidetone**, plus shows a red banner explaining the cause and the FTDI-on-CW-KEY workaround. Macros and text-send (CI-V 0x17 / `send_morse`) keep working — only paddle-element keying is gated. The misleading "tones with no RF" symptom is gone; the user knows immediately what's wrong and what to do.

A native software fix for cdc_acm specifically (a TIOCMBIS-using addon, the per-bit ioctl the driver does honor) was scoped but deferred — the audience for that exact combination is small and an external FTDI is genuinely the cleaner ham-radio solution.
