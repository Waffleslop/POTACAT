# Raspberry Pi / sBitx ALSA fix actually ships this time

**Headline:** The alsa\_native addon fix that was supposed to land in v1.6.0 was silently dropped from the build because of a syntax error in `binding.gyp`. v1.6.1 fixes the gyp file *and* adds a CI hard-fail so this kind of silent regression can't happen again.

## What happened

v1.6.0's notes promised an `alsa_native` build with the fix for HA3HZ's "undefined symbol: alloca" error. The fix was committed correctly, but the comment style I used in `binding.gyp` (`/* ... */`) broke gyp's parser — gyp uses Python's `eval()` on the file and only accepts `#` line comments. The build threw `SyntaxError: unterminated string literal`, and because `npm run build-natives` was wrapped in `continue-on-error: true` (so optional addons like FreeDV's codec2 don't block the release), the failure was swallowed. v1.6.0 arm64 shipped without the alsa addon at all.

If you downloaded v1.6.0 and ran the `node scripts/test-alsa.js` capture and got either the same "undefined symbol: alloca" or a "not available" error, that's why.

## What's fixed

- **`binding.gyp` rewritten with `#` comments.** Verified with the same parser node-gyp uses (`python -c "eval(open('binding.gyp').read())"`).
- **CI now hard-fails if `alsa_native` doesn't build.** The release workflow's Linux jobs (both x86\_64 and arm64) now have a "Confirm alsa\_native built + loads + can openCapture" step that:
  1. asserts the `.node` file exists,
  2. requires `lib/alsa.js` and checks `isAvailable()` returns true,
  3. calls `openCapture('null')` and `closeCapture` — exercises the exact code path where HA3HZ's alloca symbol error fired.
  
  If any of those fail, the release stops. No more silent shipments without the addon.

## For HA3HZ + sBitx users

Once v1.6.1 finishes building, grab `POTACAT-1.6.1-arm64.AppImage` from the release page and run:

```
./POTACAT-1.6.1-arm64.AppImage --appimage-extract
cd squashfs-root/resources/app.asar.unpacked
node scripts/test-alsa.js hw:1,1 5 out.wav
```

You should see "Capturing 5s..." followed by a `.wav` file (5 seconds of audio from whatever `hw:1,1` resolves to on your Pi) and **no** "undefined symbol: alloca" line. If you still see the alloca error, the AppImage didn't build the addon — open an issue and I'll dig further.

No other changes in this release — v1.6.0's full notes apply.
