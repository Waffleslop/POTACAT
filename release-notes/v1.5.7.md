A bigger release covering two weeks of user reports — the headline is the POTA.app sync rewrite that finally puts the recurring 403 errors to bed, plus activator dupe detection, beam-heading on VFO popouts, friendlier rig-control errors, and a stack of smaller fixes.

## POTA.app Sync — recurring 403 errors solved

Earlier versions captured the Cognito ID token at connect time and replayed it forever. Cognito ID tokens expire after one hour, so an hour after Connect every sync started 403'ing — the only workaround was Disconnect / Reconnect. This release re-architects the sync to keep the connection alive indefinitely:

- **Cookies are now the source of truth.** The Electron partition holds the Cognito session cookies (idToken, refreshToken, accessToken). Every sync reads the current id token fresh from the cookie instead of a stale snapshot in `settings.json`.
- **Refresh-token flow.** When the id token is within 60 seconds of expiry, POTACAT exchanges the refresh token at `cognito-idp.us-east-2.amazonaws.com` for a fresh id token and writes it back to the cookie. No user interaction needed. Refresh tokens stay valid for ~30 days idle, so a normal weekend operator never sees an expiry.
- **One-shot retry on 401/403.** If the API ever returns 401 or 403, POTACAT forces a token refresh and retries the same request once before surfacing an error. Catches edge cases where the server rotates keys faster than the client expects.
- **Disconnect now wipes the partition properly** — cookies, localStorage, sessionStorage, IndexedDB, and cache. The next Connect re-runs the full Cognito sign-in instead of re-using a stale SSO bounce.
- **Diagnostic logging on every sync** — token source (cookie / refreshed / refreshed-forced), expiry timestamp, and TTL in seconds. If a future report says "still 403'ing," the log tells us instantly whether it was an expiry issue or something else.
- **No JWT on disk anymore.** The `bearerToken` field is no longer written to `settings.json`. Existing copies are scrubbed on Disconnect. The token only lives in the partition cookies (where Cognito's SDK manages its lifecycle) and briefly in memory during a request.

If you've been hitting "HTTP 403 — POTA.app rejected the request" every hour, this is the fix.

## Activator Dupe Detection

Logging the same activator twice in one session is now flagged before you commit:

- **DUPE chip in the banner logger** when the callsign + park (or callsign + summit) matches a QSO already logged today.
- **Same chip on both VFO op-cards** so dual-VFO operators see it on whichever card they're working from.
- The chip is informational — you can still log a legitimate dupe (different mode/band/park) by clicking through. Pure same-band same-mode dupes are the target.

## VFO Popouts — Beam Heading

Optional bearing line under the park name on each VFO popout — distance + heading from your QTH grid to the park's reference grid. Toggle in Settings → VFO Popouts. (KM4CFT request.)

## WRL UDP Forwarding — full ADIF context

The N1MM-compatible UDP feed POTACAT sends to loggers like Win-EQF and N1MM+ now includes the full park context that ADIF would carry, not just the contest tags:

- `state` (US state from QRZ lookup)
- `sig` / `sig_info` (POTA / SOTA / WWFF / LLOTA program reference)
- `pota_ref`, `sota_ref`, `wwff_ref`, `llota_ref` for downstream parsers that read those specific fields
- For dual-program activations, all program refs forward together

(W7DB report — Win-EQF was receiving everything except the park.)

## Dual-Program Spot Logging

When a spot is announced under two programs simultaneously (POTA + SOTA, POTA + WWFF, etc.), the QSO dialog now keeps **all** program refs in the logged ADIF — previously the secondary ref was dropped, so a SOTA summit that was also a POTA park got logged with only one of the two references.

## FreeDV / Digital Voice

- **Mode now correctly identified as DIGL/DIGU** instead of USB/LSB so loggers and rig control panels render it correctly.
- **LSB FreeDV** works on lower bands.
- **Bigger Bug Report banner buttons on macOS** — they were hard to hit on the smaller bar.

## VFO Popouts — LOG Mode

New display mode for users who pop the VFO out as a dedicated logging window. Compact layout focused on QSO entry rather than tuning. Toggle per popout via the cog menu.

## ECHOCAT (Phone)

- **DX Cluster toggle now actually toggles.** Previously the phone toggle looked like it changed state but no IPC fired through to the desktop, so the underlying connection didn't open or close.
- **JTCAT Reply** — the FT8 panel can now send the highlighted QSO partner's call as the next "Reply To" without retyping.

## Rig Control

- **IC-706MKIIG added to the Icom dropdown.** Conservative caps — basic freq/mode tuning works; the rig predates the CI-V level/function sub-commands so filter switching, ATU, NB control, and remote power-on aren't wired up. Users wanting full control can fall back to rigctld (hamlib backend ID 311). (N4RDX request.)
- **rigctld errors now say what they mean.** When the rig backend returns an `RPRT -N` code, POTACAT translates it before showing it to you:
  - `RPRT -5` → "Communication timed out — radio not responding (check cable / baud rate / power)"
  - `RPRT -8` → "Protocol error — radio reply did not match expected format (wrong rig model?)"
  - `RPRT -20` → "Radio reports it is powered off — turn the radio on (or use POTACAT Rig → Power On) and try again"
  - …and a dozen others. Showing up in both the Test Connection dialog and the live CAT log.

## CW Key Port (Linux)

- **Python fallback for `cdc_acm` ENOTTY.** Some Linux kernels' `cdc_acm` driver rejects the `TIOCMSET` ioctl that node-serialport uses but accepts the `TIOCMBIS`/`TIOCMBIC` ioctls pyserial uses. POTACAT now detects ENOTTY and shells out to `python3 + pyserial` to drive DTR/RTS instead of failing silently. Requires `pyserial` (`pip install pyserial` or your distro's `python3-pyserial`).

## Cloud Sync — readable error messages

When `api.potacat.com` is briefly unreachable (502 / 503 / 504 from the gateway), the Verify Subscription / Sync Now dialogs no longer dump the raw nginx HTML body into the alert. You get a single readable line:

> POTACAT cloud is temporarily unavailable (HTTP 503). Please try again in a few minutes — if it keeps happening, email casey@potacat.com.

Cached subscription status is preserved across transient server hiccups, so a 503 won't downgrade an active supporter to "trial" in the UI.
