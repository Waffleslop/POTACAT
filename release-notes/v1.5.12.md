A focused follow-up to v1.5.11 — two SSB-over-DATA PTT bugs caught in the field, several VFO / Profiles improvements, and (finally) **Intel Mac builds in CI**.

## Intel Mac builds

POTACAT now ships a separate Intel-arch DMG alongside the Apple Silicon one. The Mac card on the download page has the primary **Apple Silicon** button with **Intel Mac (x64)** linked directly below — no more "build from source" workaround for older Macs.

Both builds are ad-hoc signed (no paid Developer ID yet), so the same `xattr` quarantine workaround at the bottom of these notes applies to either arch.

## SSB-over-DATA PTT — two related fixes

**K0OTC and KI4GT** both hit the same shape of bug under v1.5.11: PTT in voice (USB or LSB) with ECHOCAT audio active correctly switched to DIGU/DIGL... but the rig stayed in the DATA mode after PTT released instead of restoring voice. Root cause: `tune()` compared the requested mode against the last polled mode and, if a poll response didn't land between PTT-down and PTT-up, saw "you're already in USB" (the pre-PTT value, never overwritten) and skipped the restore MD command. Fix: when `tune()` actually issues a mode command, optimistically update its cached "last mode" to what we asked for. Subsequent poll responses still overwrite with reality, which is correct.

**NT0Y** (FT-710 via rigctld) found the second issue: PTT in USB at 14326.000 with ECHOCAT audio correctly switched to PKTUSB, but the rig's dial jumped to 14326.700 — TX landed 700 Hz off frequency. Initially looked like a Yaesu PSK-mode subcarrier shift, but NT0Y nailed the actual mechanism: **the Yaesu shifts the VFO by the filter-width difference** between USB (2.4 kHz) and PKTUSB (3 kHz) on every mode change. PSK menu setting is unrelated. The same drift hits anyone who QSYs band-to-band when the new band's VFO last had a different mode — e.g. 40 m USB → 20 m where 20 m last had PKTUSB for FT8. Fix: re-anchor the dial by re-sending the freq right after the mode lands. `tune()`'s M,F,M,F sandwich already covered the band-change case for rigctld users; this patches the mode-only-PTT path.

## VFO Profiles

- **ECHOCAT Profiles widget** — the desktop popout's profile list is now mirrored on the phone. New opt-in widget on Full VFO View, bidirectional sync over WebSocket. Tap a profile on the phone → rig tunes; save / delete on either side updates both UIs and persistence.
- **Inline rename via edit button** — a pencil icon next to the delete X on every profile. Click → name becomes an input prefilled with the current name → Enter commits, Esc cancels. Works in both the desktop popout and the phone widget.

## Spots & Panadapter

- **Panadapter source filtering, independent from the Spot Table** — new Settings → Panadapter & Bandscope section with a "Sync with Spot Table View" toggle and nine per-source picks (POTA, SOTA, WWFF, LLOTA, DX Cluster, RBN, CW Spots, PSK Reporter, WSJT-X). Unsynced, the panadapter has its own allowlist driving SmartSDR / Thetis / Bandscope overlays. (Driven by K0OTC's request for a DX-only panadapter while keeping the table on POTA only.)
- **Split-mode mystery toggle, fixed** — KQ4MHD reported split mode turning on across sessions with no obvious cause. Root cause: a single-letter `S` hotkey toggled split, and clicking any button left it focused, so a stray `s` keystroke flipped split — which then persisted to `settings.json` and bit the next session's first tune. **The S hotkey is gone**; toggle is now Settings → Spots only. New status-bar **SPLIT** pill (amber, hidden by default, click to disable) so split is always visible while it's on.

## JTCAT FT8

- **Sequence-aware double-click reply** — Chris N4RDX: double-clicking a stale or mid-QSO message used to roll the QSO back to the start. The old parser collapsed step 3 (plain `-10`) and step 4 (`R-10`) into one path — on step 4 we'd send another R+report instead of advancing to RR73. Both renderers (popout + phone) now classify received messages into an explicit next-step tag (`reply-cq` / `send-report` / `send-r-report` / `send-rr73` / `send-73`) and main.js routes off that. Now: I CQ → reply → report → no answer → minutes later their R+SNR comes in → double-click sends RR73, not another R-report.

## CW

- **Paddle works again on rigctld with a dedicated CW Key Port** — G5HOW (Win11, FTDx10 via Hamlib): paddle was working in v1.5.3, broken by v1.5.11. The April rigctld-paddle-refusal fix for KM4CFT (silent TX) was returning early from the keyer-output callback *before* the dedicated CW Key Port handler. Users on rigctld with a separate FTDI on the rig's KEY jack lost both paths. Now: skip cat-side keying methods on rigctld, but fall through to `cwKeyPort`. Lazy-open the dedicated key port on first paddle event regardless of `paddleKey` capability, so Yaesu rigs with `paddleKey: 'txrx'` still get the FTDI path. Phone-side `cw-paddle-available` signal extended so the FTDI-workaround banner shows correctly when rigctld refuses *and* no dedicated port is configured.

## Cloud Sync

- **Settings → Cloud no longer flickers** — the cloud-status display was flashing multiple times per second whenever the tab was open, with QSO counts, device counts, and Last Sync alternating "just now"/"never". Caused by a MutationObserver watching the entire settings dialog with `subtree:true` for class changes — every refresh modified internal classes, which retriggered the observer, which re-ran refresh. Now scoped to just the dialog's `open` attribute and the cloud fieldset's own `class` (the tab-switch toggle), so internal status mutations don't loop.
