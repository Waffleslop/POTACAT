# FT8 Timing Rebuilt, DX Cluster Spots Land on the Map

Headline: the FT8 TX scheduler is rebuilt on a polling-tick architecture that ends the slot-misclassification bug class (the "transmits every 15 s instead of every 30 s" failure), DX cluster spots now plot at real map positions instead of all stacking on one country-centroid pixel, the Mode filter dropdown is cleaned up, and three crash/regression fixes from this week's testing round are folded in.

If you run FT8 from ECHOCAT iOS, this is the build where QSOs reliably complete.

## FT8 — TX scheduler rebuilt as a polling tick (Casey K3SBP)

The FT8 engine's TX and decode scheduling used two `setTimeout`-based one-shot timers that each fired at a slot boundary and made a slot-parity decision at that exact instant. The problem: Node's `setTimeout` is allowed to fire up to ~1 ms *before* its scheduled time, so a timer aimed at wall `:45.000` could land at `:44.999` — still inside the *previous* slot by the floor-division math. The engine then evaluated the wrong slot parity, the locked-slot check passed against it, and audio bled into the responder's slot. The visible symptom was an FT8 QSO transmitting the same message every 15 seconds (both slots) instead of every 30, never decoding a reply because POTACAT was drowning out the responder in their own slot.

v1.5.21 replaces both one-shot timers with a single 100 ms polling tick — the same architecture WSJT-X's `guiUpdate` loop uses. Each tick re-reads wall clock, computes the current slot, and decides whether decode or TX needs to fire, deduplicated so each event fires at most once per slot. A timer landing a millisecond early is no longer a problem: the next tick, well inside the correct slot, catches the work. The entire "fired at the wrong instant" failure class is gone.

Slot-parity evaluation also picked up a 500 ms look-ahead matching the renderer's leading-silence pad, so the slot is computed for where the audio will actually land on the air — belt-and-suspenders with the polling tick.

One smaller FT8 fix rode along: when the engine starts mid-slot, the first cycle-boundary used to send a buffer that was 80%+ leading zeros to the decoder, which predictably returned 0 decodes and burned ~10 ms of native CPU for nothing. That partial first cycle is now skipped (the silence watchdog still runs, so a genuinely dead capture is still caught).

## DX Cluster spots now land on real map positions (Casey K3SBP)

DX cluster spots carry only a callsign, frequency, and comment on the wire — no grid, no coordinates. POTACAT was resolving position from cty.dat, which returns one DXCC-entity centroid per country. Every US DX spot landed on the same pixel in Kansas, every German spot on the same pixel near Frankfurt — the map view was unreadable for cluster traffic.

Two-stage fix:

- **Call-area centroid fallback.** The synchronous spot builder now consults the call-area coordinate table (the same one the RBN map already uses) before falling back to cty.dat. A `W6` spot lands in California, `W1` in New England, `JA8` in Hokkaido, `VK2` in NSW. Countries not in the table still get the cty.dat centroid, which is fine for visually distinguishing them.
- **QRZ grid refinement.** When QRZ is configured, every new cluster spot fires a background QRZ lookup. If the operator's grid comes back, the spot's position is upgraded from the call-area centroid to the actual Maidenhead grid square and the map updates within ~200 ms. QRZ's existing cache means the lookup only hits the network once per unique callsign per session. With QRZ not configured, the call-area fallback alone already produces a reasonable map.

Distance and bearing now compute for call-area-precision coordinates too — intra-country distances (New England to California) are meaningful now, where the old cty.dat-centroid-only path skipped them as Kansas-to-Kansas noise.

## Mode filter dropdown — cleaned up (Casey K3SBP)

The Mode filter dropdown is reordered and trimmed: **All, Radio, SSB, CW, FT8/4/2, FM, FreeDV, JS8, RTTY, PSK31**.

- FT8, FT4, and FT2 are collapsed into one combined **FT8/4/2** entry — one checkbox covers all three FT-family modes.
- The **Digital (all)** preset and the **Unknown / no mode listed** entry are removed.
- Spots that arrive with no mode (POTA activators who only spotted a frequency) now always pass the mode filter instead of needing the removed "unknown" bucket — they no longer vanish when you filter to a specific mode.
- Saved filter selections from older builds (individual FT8/FT4/FT2 or "unknown") are migrated automatically on first launch — no manual re-selection needed.

## Crash and regression fixes from this week's test round

**Main-process crash on iOS tune with WebSDR active.** The iOS app sends the tune frequency as a JSON string; the KiwiSDR client called `.toFixed()` on it and threw an uncaught `TypeError`, crashing the whole desktop process. Only triggered when an iOS tune request landed while a KiwiSDR/WebSDR receiver was active. The frequency is now coerced to a number at the WebSocket protocol boundary, with malformed values rejected outright.

**CW sidetone never reached ECHOCAT iOS.** v1.5.20 shipped synthesized CW sidetone for the iOS audio bridge, but the same release's WinKeyer-priority fix added early `return` statements to the CW-send dispatch — and the sidetone trigger lived *after* those returns. Every Flex user (the main ECHOCAT CW audience) hit a `return` before the sidetone fired. The sidetone trigger and the audio-health TX lockout are now hoisted above the dispatch branches, so they run regardless of which keyer backend handles the send. If you're on the iOS audio bridge and send a CW macro, you'll hear it now.

**Blank spots table.** A dropdown-cleanup pass left a call to a function that had been removed in the same change; the resulting startup error halted the renderer before the spots table rendered. Fixed — caught and resolved during testing, but noting it here since a dev build briefly carried it.
