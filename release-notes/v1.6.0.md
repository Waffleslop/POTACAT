# The Whole Lotta Stuff is Working Flawlessly

**Headline:** Elecraft K4/K4D remote-over-the-network lands end to end — encrypted CAT *and* Opus audio over a single TCP connection, no DAX-equivalent middleware required. Plus a full Solar &amp; Propagation Conditions popout (its own resizable window, zoomable, theme-aware), user-defined net reminders that finally sync to the phone, and the Raspberry Pi ALSA addon fix HA3HZ has been chasing for a week.

## Elecraft K4 / K4D over the network — RX, TX, CAT, audio, the works

Up to this release the K4 was a serial-only rig in POTACAT. v1.6.0 ships the full network stack, in four phases now shipped together:

- **Phase 1 — framed CAT + SHA-384 auth on port 9205.** Connect to a K4 anywhere on your network (or anywhere on the public internet) using the same protocol the official K4 Remote does. Frame format is `FE FD FC FB | len:BE32 | payload | FB FC FD FE`; auth handshake is SHA-384 over the password plus a server-issued salt.
- **Phase 2 — UI for the new connection type.** Settings → Radio → "K4 Remote (network)" with host / port / password fields. New rig connections persist alongside your serial K4s; switch with one click.
- **Phase 3 — TLS-PSK on port 9204 (recommended for the public internet).** Same SHA-384 password, but tunnelled through TLS using `tls.connect({pskCallback})`. CAT and (now) audio frames are both encrypted; no plain-text password ever crosses the wire.
- **Phase 4 — Opus audio over the same TCP socket.** RX audio is 12 kHz stereo Opus (L = MAIN, R = SUB) decoded by the addon-grade `@discordjs/opus` codec. TX audio (your mic via the ECHOCAT bridge or a voice macro) is re-encoded as 12 kHz mono and pushed back over the same socket. Frames carry a 7-byte header `[type:1][ver:1][seq:1][mode:1][frameSize:LE16][srCode:1][data...]`. PTT is the existing TX/RX command — no special audio gating.

What you can do now that you couldn't before:
- Operate your K4 from anywhere on the public internet without a VPN.
- Decode FT8 directly from the K4 over the wire — JTCAT taps the same 12 kHz mono stream `handleK4AudioFrame` decodes, no Windows audio device at all.
- Run SSTV with the K4 as the audio source (4× upsampled to 48 kHz on the way to the decoder + waterfall popout, same pattern as SmartSDR Direct).
- Send voice / FT8 over WebRTC from your phone → desktop → K4, end-to-end, with TLS encryption on every hop you don't control.

The plain-TCP path on port 9205 is still there for trusted LANs; pick whichever fits your threat model.

## Conditions popout — solar &amp; propagation in its own window

**More → Conditions ↗** opens a new resizable window with the full solar and propagation picture. Built from two free, no-API-key feeds:

- **hamqsl.com (N0NBH solarxml.php)** — SFI, sunspot number, A, K, X-ray flare class, 304 Å EUV, proton/electron flux, MUF, foF2, signal noise, geomag field text, Bz, solar wind, aurora activity + southernmost limit, and the full HF day/night band-conditions matrix plus VHF / E-skip status.
- **NOAA SWPC** — Kp index sparkline (last 24 hours, 3-hour samples) and recent space-weather alerts (last 24 hours, severity-tinted).

Six color-coded cards in a responsive grid:
- **Solar Activity** — hero tiles for SFI / SN / X-Ray / 304 Å with severity coloring (green / yellow / red) plus a KV strip for MUF / foF2 / flux / signal noise / normalization.
- **Geomagnetic** — hero tiles for K / A / SW Speed / Bz plus field state / aurora / aurora limit.
- **HF Bands** — band-by-band Day / Night Poor-Fair-Good pill chips.
- **VHF / E-Skip** — phenomenon × region × open/closed.
- **Kp last 24 h** — gradient-bar sparkline with current + 24h peak readout.
- **NOAA SWPC Alerts** — last 24 hours, one-line truncated, hover to expand.

Every hero tile has a hover tooltip explaining what the number means and what threshold matters ("SFI &gt;120 supports good HF DX; &lt;90 is weak F-layer," "strong negative Bz couples with Earth's field and triggers storms"). The status-bar SFI / K / A pills you've always had still update from the same feed — the popout is the deep dive when you want it.

The window:
- Remembers its size and screen position across closes (`settings.conditionsPopoutBounds`).
- Remembers its zoom level — Ctrl/Cmd + = / − / 0 or Ctrl/Cmd + mouse wheel, persists across reopens.
- Tracks the main app's light/dark theme live (severity palette swaps to dark amber on light mode so the warn yellow doesn't disappear into white).
- Esc closes.

## User-defined net reminders sync to ECHOCAT / mobile Dir tab

If you've added entries to **Settings → Notifications → My Net Reminders**, those nets now appear on the phone's **Dir** tab alongside the community list — with a small "MY" badge so you can tell yours apart. The `directory` push to ECHOCAT clients carries a new `userNets` field; mobile already shipped the consumer (NetEntry.isUser flag + (name, frequency) dedupe against community), so this commit alone lights up the feature.

Live sync: edit the desktop reminder's frequency / time / Enabled flag and the phone reflects the change within seconds — no app restart, no manual refresh. Disabled reminders are filtered out before the push; uncheck Enabled and the row drops on the next tick.

## Raspberry Pi: alsa_native addon load no longer crashes ("undefined symbol: alloca")

HA3HZ on a Pi Zero 2 running the v1.5.25 arm64 AppImage hit:

```
node: symbol lookup error: alsa_native.node: undefined symbol: alloca
```

at the moment `scripts/test-alsa.js` called `openCapture`. The addon `.node` loaded fine; the alloca symbol was lazy-bound and only resolved on first use.

Two compounding root causes, both fixed:
1. `binding.gyp` set `-std=c11` (strict C11, no GNU extensions). In that dialect glibc's `<alloca.h>` declares alloca only as an extern prototype — the compiler emits a real function call, and at runtime libc doesn't always export alloca as a real symbol on every distro. Switched to `-std=gnu11 -D_GNU_SOURCE` so the GCC alloca builtin is always available.
2. The addon used `snd_pcm_hw_params_alloca(&hw)`, a macro that expands to a raw `alloca()` call. Replaced with `snd_pcm_hw_params_malloc` + `_free`. Heap-allocated, freed on every error path and right after a successful commit — no compiler-dialect dependency at all.

The `linux-validate` workflow's smoke test was also bolstered to call `openCapture('null')` so any future regression in this class is caught at validate time rather than at the user's first capture attempt.

## Other improvements rolled up

- **HF privilege tables** updated: 60 m channelized + 33 cm / 23 cm bands added, sub-band splits added for 30 m / 17 m / 12 m. 60 m CH3 typo fixed (5366.6 → 5366.5) across us\_extra / advanced / general.
- **SSTV waterfall** now renders on SmartSDR Direct — VITA-49 audio is 4× upsampled to 48 kHz and routed to both the decoder and the popout's FFT (same pattern as the FT8 fix in v1.5.25).
- **K4 SSB tune** no longer flips to DATA-A mid-cycle. The `_digiMd` override was using `mapped.da != null` which matched voice modes (da = 0); changed to `mapped.da > 0`. N7QT reported.
- **K4 network: XIT enabled on tune** — `cat.hasNativeXit` is now set for the `k4-network` connection type, so the existing CW XIT offset takes effect.
- **CW sidetone for non-Flex rigs** — Icom / Yaesu / Kenwood now get a sidetone when the desktop sends CW via the keyer (was Flex-only before).
- **Audio bridge silent-restart false positive** suppressed on quiet bands. The 5 s peak-zero threshold was too tight; bumped to 90 s and disabled entirely under SmartSDR Direct (which has its own frame-stall detector).
