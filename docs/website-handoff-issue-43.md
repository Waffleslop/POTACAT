# Website Handoff — Issue #43: Linux download serves arm64 / add both arches

**Repo to change:** `potacat-website` (the marketing site — separate from the POTACAT app repo).
**GitHub issue:** https://github.com/Waffleslop/POTACAT/issues/43
**Reported:** Linux download button downloads the arm64 build instead of amd64; please offer both as buttons.

---

## TL;DR for the website agent
1. The download block in `index.html` (between the `<!-- DOWNLOAD-START -->` / `<!-- DOWNLOAD-END -->` markers) is **stale** — its Linux link almost certainly points at the wrong/old asset. Regenerate it.
2. The block must offer **both x86_64 and arm64** Linux downloads, with the arch clearly labeled.
3. Canonical way to do this: run the app repo's generator script against the site's `index.html` (see below). If you can't, hand-patch the Linux card with the exact HTML in this doc.

---

## Root cause (verified against the v1.8.13 release)
- The **unsuffixed AppImage `POTACAT-<version>.AppImage` IS the x86_64 build.** electron-builder only adds an arch suffix for non-x64, so arm64 is `POTACAT-<version>-arm64.AppImage`. Confirmed by listing the actual release assets.
- So if the live site downloads arm64 from the main Linux button, the live `index.html` is out of sync with the current generator (hand-edited, or generated by an older script). **Check the live link** — if its `href` ends in `-arm64.AppImage`, that's the bug.
- Separately, the site offered **no arm64 download at all**, even though every release ships arm64 AppImage/.deb/.rpm for the Raspberry Pi / sBitx / ARM SBC audience. That's the "add both as buttons" request.

---

## Source of truth — exact Linux asset names (every release follows this pattern)
Asset URL base: `https://github.com/Waffleslop/POTACAT/releases/download/v<VERSION>/<NAME>`

| Arch | AppImage | .deb | .rpm |
|------|----------|------|------|
| **x86_64** (Intel/AMD) | `POTACAT-<VERSION>.AppImage` | `potacat_<VERSION>_amd64.deb` | `potacat-<VERSION>.x86_64.rpm` |
| **arm64** (aarch64, Pi/SBC) | `POTACAT-<VERSION>-arm64.AppImage` | `potacat_<VERSION>_arm64.deb` | `potacat-<VERSION>.aarch64.rpm` |

(Windows/Mac assets are unchanged; this issue is Linux-only.)

---

## Preferred fix: regenerate with the canonical script
The POTACAT **app** repo already contains the generator that owns this download block — fixed in commit `078ebbd` to emit both arches. Run it against the website's `index.html`:

```bash
# from a checkout of the POTACAT app repo (Waffleslop/POTACAT)
node scripts/update-website-links.js <VERSION> /path/to/potacat-website/index.html
```

It replaces everything between `<!-- DOWNLOAD-START -->` and `<!-- DOWNLOAD-END -->`, so it's idempotent and safe to re-run. This is the correct long-term path — the live site should always be generated from that script so it never drifts again.

## Notes / gotchas
- Don't invent asset names — copy them from the table above or the live GitHub release assets list. The three naming conventions differ per package type (`amd64` for .deb, `x86_64` for .rpm, no-suffix for AppImage; `arm64` for .deb/AppImage, `aarch64` for .rpm).
- This is **website-only**. No POTACAT app code change is needed — the app already builds and publishes all six Linux artifacts.
